05 / 11กรณีศึกษาที่ 1 · แนวทางจัดการ

FCAPS - Process – Method – Procedure – Task – Tools

FCAPSการจัดการด้านความผิดพลาด (Fault Management) และด้านประสิทธิภาพ (Performance Management) ของเครือข่าย
Processค้นหาสาเหตุที่ทำให้ ERP ทำงานช้าช่วงบ่ายและระบบล่มเป็นบางวัน พร้อมลดข้อผิดพลาดในการส่งแพ็กเก็ตของ switch หลัก เพื่อให้ผู้ใช้ทั้ง 200 คนใช้งานได้ต่อเนื่องและมีเสถียรภาพ
Methodตรวจสอบ log ของ ERP Server และ switch หลัก ตรวจสถิติ error ของพอร์ต (CRC error, packet drop, collision) และ monitor ปริมาณ traffic ช่วงเวลาที่เกิดปัญหา เทียบกับช่วงปกติ
Procedureรวบรวม log → ตรวจสถิติ error บน switch ย้อนหลัง → monitor bandwidth/traffic ช่วงบ่าย → ทดสอบด้วย ping/traceroute → วิเคราะห์สาเหตุ (สาย/พอร์ตชำรุด, bandwidth ไม่พอ, switch เสื่อม) → ดำเนินการแก้ไข
Taskทีมผู้ดูแลระบบเครือข่ายปฏิบัติตามลำดับ procedure ทีละขั้น พร้อมบันทึกผลการตรวจสอบและการแก้ไขใน Incident Report เพื่อติดตามผล
ToolsSNMP Monitoring (PRTG/SolarWinds) · Wireshark · Switch Management Console (CLI/Web GUI) · คำสั่ง ping, traceroute · Log Analyzer / Syslog Server

08 / 11กรณีศึกษาที่ 4 · แนวทางจัดการ

FCAPS - Process – Method – Procedure – Task – Tools

FCAPSการจัดการด้านการบันทึก/บัญชีการใช้งาน (Accounting Management) และด้านประสิทธิภาพ (Performance Management) ของเครือข่าย
Processเก็บรวบรวมข้อมูลปริมาณการใช้แบนด์วิดท์ของแต่ละแผนก เพื่อจัดทำรายงานเสนอฝ่ายบริหารสำหรับปรับปรุงนโยบาย QoS และกำหนดโควต้าการใช้งานแอปพลิเคชัน เช่น video streaming
Methodแบ่ง VLAN / IP Address ตามแผนก ใช้เครื่องมือ monitor traffic แบบ NetFlow/sFlow และ SNMP เก็บสถิติการใช้งานย้อนหลังตามช่วงเวลา แล้ววิเคราะห์สรุปเป็นรายงาน
Procedureกำหนด VLAN/IP แยกตามแผนก → ติดตั้งและกำหนดค่าเครื่องมือ monitor traffic → เก็บรวบรวมข้อมูลการใช้งานต่อเนื่อง → วิเคราะห์และจัดทำรายงานรายแผนก → เสนอปรับปรุงนโยบาย QoS และโควต้า
Taskฝ่าย IT จัดทำรายงานการใช้แบนด์วิดท์เป็นประจำ (รายสัปดาห์/รายเดือน) และนำเสนอต่อผู้บริหารเพื่อพิจารณาอนุมัตินโยบาย QoS และโควต้าการใช้งาน
ToolsNetFlow / sFlow Analyzer · SNMP Monitoring (PRTG/SolarWinds) · Router/Switch QoS Configuration · Bandwidth Monitoring Dashboard · Excel / เครื่องมือจัดทำรายงาน

สองกรณีศึกษา สองมุมของการจัดการเครือข่าย

กรณีศึกษาที่ 1

ERP ช้า / ล่ม — Fault + Performance

เริ่มจากอาการผิดปกติ (fault) ที่กระทบประสิทธิภาพ (performance) ต้องอาศัยการวิเคราะห์ log และเครื่องมือ monitoring เพื่อหาสาเหตุที่ต้นทางแล้วแก้ไข

กรณีศึกษาที่ 4

รายงานแบนด์วิดท์ — Accounting + Configuration

เริ่มจากความต้องการเชิงบริหาร (accounting) เพื่อวางนโยบายการใช้ทรัพยากรอย่างเป็นธรรม แล้วนำไปสู่การปรับตั้งค่า (configuration) QoS/โควต้า
